# Mock Product Seeding Threat Model

## Scope

Sprint 6 synthetic product generators, seed script, generated manifest and local
asset tree.

## Assets

- Product metadata and ACG assignments.
- Access scenario definitions used by later search and UI work.
- Generated asset bytes and SHA-256 hashes.
- Local output paths selected by developers or CI.

## Threats And Controls

| Threat | Control in Sprint 6 |
|---|---|
| Real intelligence content is accidentally committed as seed data. | Generators create fictional products only, every product and asset carries `MOCK DATA ONLY`, and generated binary assets remain outside Git by default. |
| Seed output path writes outside the intended local directory. | Asset paths are generator-controlled relative paths beneath `assets/<product-reference>/`; tests assert no absolute paths or parent traversal. |
| Downstream search or access tests use inconsistent ACG data. | ACG definitions and named access scenarios are emitted in the manifest with deterministic product assignments. |
| Hashes or IDs change unexpectedly and mask regressions. | IDs use a fixed UUID namespace and hashes are derived from deterministic generated bytes. |
| New generator dependencies introduce avoidable supply-chain risk. | Sprint 6 uses only the Python standard library for mock generation. |
| Mock imagery or document structure is mistaken for operational material. | Asset contents are deliberately simple placeholders and include explicit mock markers. |

## Deferred Risks

- Real upload malware scanning and MIME validation remain deferred until the
  upload/object-storage sprint.
- Database import, object-store permissions and access-controlled downloads are
  handled by later ingestion work.
- Search-result leakage must be rechecked when Sprint 7 indexes this generated
  catalogue.
